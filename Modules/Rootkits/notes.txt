ARCHITECTURE DE HOOKING (évolution)
------------------------------------
Tentative 1: Syscall table patching (CR0.WP)
  → CRASH sur machine physique AMD: CR4.SMAP=1 interdit effacement CR0.WP
    (AMD APM vol.2 §8.4.3)

Tentative 2: set_memory_rw() via kallsyms
  → Non exporté / échoue silencieusement

Tentative 3: lookup_address() + PTE _PAGE_RW flip (mapping logique)
  → CONFIG_STRICT_KERNEL_RWX: écriture toujours bloquée

Tentative 4: phys_to_virt() + écriture direct mapping
  → Direct mapping aussi RO sur noyaux récents

Tentative 5: Hybride CR0.WP + SMAP detection
  → Écriture RÉUSSIT (confirmé via /proc/kcore), mais hooks JAMAIS APPELÉS
  → CAUSE: Linux 6.1 + CONFIG_MITIGATION_SPECTRE_BHI=y remplace le dispatch
    sys_call_table[nr](regs) par x64_sys_call(regs,nr) — une fonction compilée
    avec des appels DIRECTS (non-indirects) vers chaque handler. La table est
    patchée mais jamais lue pour le dispatch.

Solution finale: KRETPROBES
  → register_kretprobe() sur __x64_sys_read, __x64_sys_getdents64, __x64_sys_kill
  → Fonctionne sur TOUS kernels (4.x+), survit à toutes les protections syscall table
  → entry_handler sauvegarde les arguments; handler modifie la valeur de retour
